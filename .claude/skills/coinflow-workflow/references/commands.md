# 三个命令

## 命令 1：`/plan` — 需求讨论

**用途**：基于当前项目整体功能及架构，讨论需求方向，不进入开发。

**流程**：
1. 阅读 `CoinFlow技术架构文档.md` 了解当前架构边界
2. 阅读 `docs/PROJECT_PLAN.md` 了解已完成功能与进行中需求
3. 与用户探讨需求可行性、影响范围、优先级
4. 产出讨论结论，更新 `docs/PROJECT_PLAN.md`「待开始需求」表

**进入条件**：用户提出模糊想法或方向性需求，尚未形成明确的功能边界。

**产出**：PROJECT_PLAN.md 中待开始需求记录。

**不进开发**：此阶段仅讨论和规划，不做 BA/DEV/QA 流程，不创建 feature 文件夹。

---

## 命令 2：`/feature` — 新需求开发

**用途**：将明确需求经过 BA→DEV→QA 流程完整交付。

**进入条件**：需求已经过 `/plan` 讨论，明确了边界和优先级。

**三角色流程**：

```
用户确认需求
    │
    ▼
┌─────────────────────────────────────────────┐
│ Step 0: 历史检查（必须）                       │
│                                              │
│ 1. 扫描 docs/features/ 已有需求               │
│ 2. 扫描 docs/bugfixes/ 相关模块历史 Bug        │
│ 3. 阅读 PROJECT_PLAN.md 确认无功能冲突         │
│ 4. 如有冲突/重叠 → 与用户讨论处理策略再继续     │
└──────────────┬──────────────────────────────┘
               │ 无冲突
               ▼
┌─────────────────────────────────────────────┐
│ Role 1: BA（需求澄清者）                      │
│                                              │
│ 使用 /brainstorming skill 与用户探讨：        │
│   - 需求边界与范围                            │
│   - 用户故事与验收标准                         │
│   - 依赖与风险（引用历史检查发现的关联项）       │
│                                              │
│ 产出: docs/features/<name>/BA/requirements.md │
└──────────────┬──────────────────────────────┘
               │ BA 完成
               ▼
┌─────────────────────────────────────────────┐
│ Role 2: DEV（开发者）                         │
│                                              │
│ 1. 阅读 BA/requirements.md                   │
│ 2. 撰写技术设计文档，验证可行性                │
│ 3. 严格遵循技术边界 + andrej-karpathy 编码原则 │
│ 4. 编写代码 + 单元测试（80%+ 覆盖）            │
│                                              │
│ 产出: docs/features/<name>/DEV/              │
│       ├── tech-design.md                     │
│       └── implementation-notes.md            │
└──────────────┬──────────────────────────────┘
               │ DEV 完成
               ▼
┌─────────────────────────────────────────────┐
│ Role 3: QA（测试者）                         │
│                                              │
│ 1. 阅读 BA/requirements.md + DEV/tech-design.md │
│ 2. 编写测试用例                               │
│ 3. 真机模拟测试（必须真机，iOS 26.x）          │
│ 4. 发现问题 → 提回 DEV 修复 → 重新验证         │
│ 5. 循环直到全部通过                           │
│ 6. 通知用户验收                               │
│                                              │
│ 产出: docs/features/<name>/QA/               │
│       ├── test-cases.md                      │
│       └── test-report.md                     │
└──────────────┬──────────────────────────────┘
               │ QA 全部通过
               ▼
          用户验收 → 更新 PROJECT_PLAN.md
```

## 命令 3：`/bugfix` — Bug 修复

**用途**：根据用户描述修复问题，走 DEV→QA 流程（无 BA 阶段）。

**流程**：

```
用户报告 Bug
    │
    ▼
┌─────────────────────────────────────────────┐
│ Step 0: 历史检查（必须）                       │
│                                              │
│ 1. 搜索 docs/bugfixes/ 同模块/同名历史修复     │
│ 2. 如找到 → 阅读历史 root-cause.md            │
│ 3. 判断是否为回归或同一根因                    │
│ 4. 如可复用历史方案 → 直接引用，跳过新分析     │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Role 2: DEV                                 │
│                                              │
│ 1. 复现问题 + 根因分析（如为回归则引用历史）    │
│ 2. 设计修复方案                               │
│ 3. 编码修复 + 自测                            │
│                                              │
│ 产出: docs/bugfixes/<name>/DEV/              │
│       ├── root-cause.md                      │
│       └── fix-notes.md                       │
└──────────────┬──────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────┐
│ Role 3: QA                                  │
│                                              │
│ 1. 编写回归测试用例                           │
│ 2. 真机验证修复 + 回归测试（iOS 26.x）         │
│ 3. 不通过 → 提回 DEV                         │
│ 4. 通过 → 通知用户确认                        │
│                                              │
│ 产出: docs/bugfixes/<name>/QA/               │
│       ├── test-cases.md                      │
│       └── verification-report.md             │
└──────────────┬──────────────────────────────┘
               │
               ▼
          用户确认 → 更新 PROJECT_PLAN.md
```

## 命令 4：`/polish` — UI 调整与细节优化

**用途**：根据用户反馈对现有功能进行 UI/UX 改造与细节打磨，不走需求分析（BA），从视觉与交互层面先设计再开发。

**进入条件**：用户提出「优化 xx 展示」「调整 xx 样式」「xx 不太好看」等 UI/UX 类反馈。

**三角色流程**：

```
用户反馈 UI/UX 问题
    │
    ▼
┌─────────────────────────────────────────────┐
│ Role 1: UI（视觉设计师）                       │
│                                              │
│ 1. 阅读当前模块代码，理解现有 UI 结构            │
│ 2. 参考行业通用方案（iOS HIG、同类 App）         │
│ 3. 设计改造方案，给出详细规范：                  │
│    - 布局（间距/对齐/层级）                      │
│    - 样式（颜色/字体/圆角/阴影/动效）             │
│    - 交互（点击/滑动/状态转换/反馈）              │
│    - 主题适配（三主题兼容性）                    │
│ 4. 与用户确认方案（可多方案对比）                 │
│                                              │
│ 产出: docs/polish/<name>/UI/design-spec.md    │
└──────────────┬──────────────────────────────┘
               │ UI 方案确认
               ▼
┌─────────────────────────────────────────────┐
│ Role 2: DEV（开发者）                         │
│                                              │
│ 1. 阅读 UI/design-spec.md                    │
│ 2. 按规范实现 UI 改造                         │
│ 3. 三主题验证（Notion/Liquid Glass/Animal Island）│
│ 4. 单元测试（80%+ 覆盖）                       │
│                                              │
│ 产出: docs/polish/<name>/DEV/                │
│       ├── implementation-notes.md            │
│       └── 代码改动                            │
└──────────────┬──────────────────────────────┘
               │ DEV 完成
               ▼
┌─────────────────────────────────────────────┐
│ Role 3: QA（测试者）                         │
│                                              │
│ 1. 逐条对比开发结果与 UI 规范                  │
│ 2. 截图对比（before/after）                    │
│ 3. 三主题全量测试                             │
│ 4. 不通过 → 标差异项 → 提回 DEV               │
│ 5. 循环直到全部符合 UI 规范                    │
│ 6. 通知用户最终验收                            │
│                                              │
│ 产出: docs/polish/<name>/QA/                 │
│       ├── visual-diff.md                     │
│       └── test-report.md                     │
└──────────────┬──────────────────────────────┘
               │ QA 全部通过
               ▼
          用户验收 → 更新 PROJECT_PLAN.md
```

### UI 角色的设计规范（必须覆盖）

| 维度 | 内容 |
|------|------|
| **布局** | 组件间距、对齐方式、层级结构（z-index/overlay）、响应式策略 |
| **颜色** | 前景/背景/边框色值、深色/浅色映射、主题感知 token |
| **字体** | 字号阶梯、字重、行高、字符间距（monospaced/proportional） |
| **形状** | 圆角半径、阴影（offset/radius/opacity）、边框宽度 |
| **动效** | 过渡类型（spring/ease/linear）、时长、延迟、触发条件 |
| **交互** | 按压反馈（haptic/scale/color）、手势响应区域、加载/空/错误态 |
| **主题** | Notion / Liquid Glass / Animal Island 三主题适配 |

### /polish 与 /feature 的区别

| | `/feature` | `/polish` |
|---|-----------|----------|
| 起点 | 功能需求 | 视觉/交互反馈 |
| 第一角色 | BA（需求澄清） | UI（视觉设计） |
| 核心产出 | requirements.md + tech-design.md | design-spec.md（含布局/样式/动效规范） |
| 改动范围 | 跨多层（数据+UI+统计） | 集中在 View 层 + Theme 层 |
| 历史检查 | 扫描 features/ + bugfixes/ | 扫描 polish/ 历史避免重复改造 |
